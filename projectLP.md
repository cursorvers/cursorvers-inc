以下、そのまま **Claude Code（Cursor内のClaude Code想定）に貼って使える「指示書」**です。
目的は「文章を刺さる形に差し替えつつ、将来の拡張（監査SaaS/教育/新メニュー追加）に耐える構造にする」こと。
コード側は **“文言の一元管理（content定義）”**まで含めて指示しています。

---

```text
# Cursorvers LP コピー改善：Claude Code 指示書（コード更新）
あなたはNext.js + Tailwind（shadcn/ui含む）プロジェクトのエンジニアです。
目的は、LPの文章を「刺さるコピー」に更新しつつ、将来の拡張性を保つために
文言をコンポーネント直書きではなく “content定義に集約” することです。

## 0) ゴール（Outcome）
- LPの主要セクションの見出し・本文・CTA文言を指定の新コピーに差し替える
- 文言は `content/landing.ts`（または既存のcontent管理箇所）に集約し、各セクションは参照する形にする
- レイアウト崩れ（SP/PC）・改行不自然・ボタン幅不足が起きないように調整
- 断定的な医療/法規の保証表現は避け、「目指す」「整備する」等の状態表現にする
- lint/buildが通る状態でコミット

## 1) 重要な制約（守る）
- 既存のデザイン（配色・余白・コンポーネント）を大きく変えない。文章差し替えが主目的
- H1の「AIに、臨床の魂を。」は維持（ブランド資産なので残す）
- ただし、Hero直下の説明文/CTAは新コピーへ
- 文章は “拡張の背骨” として「ルール × ログ × 教育」を核にする（複数セクションで整合）
- 「完全準拠」「絶対安全」「必ず」など保証に見える言い回しは使わない

## 2) 作業手順（高レベル）
1. LPの該当ページと各セクションコンポーネントを特定
   - App Routerなら `app/page.tsx` や `app/(marketing)/page.tsx` など
   - セクションが分割されていれば `components/sections/*` を探索
2. 既存文言で ripgrep 検索して該当箇所を特定（例）
   - "AIに、臨床の魂を"
   - "知的資産"
   - "Empower"
   - "AIガバナンスの民主化"
   - "セルフ" "3分"
   - "48h" "48H" "簡易診断"
   - "なぜCursorvers"
   - "3 Core Business Axes"
   - "顧問パートナー"
3. 文言の一元管理ファイルを新設 or 既存に追記
   - 推奨：`src/content/landing.ts`（または `content/landing.ts`）
   - export const landingCopy = { hero: {...}, sections: {...} } の形
4. コンポーネント内直書きの文字列を、content参照に置換
5. 新コピーに差し替え（指定文言は後述）
6. 表示確認（SP/PC）
   - 長文でカードがはみ出す/ボタンが折り返す場合：
     - `max-w` / `leading` / `text-balance` / 改行位置（段落分割）を調整
7. lint / typecheck / build を実行し、通ることを確認
8. コミット（例：`feat: update landing copy and centralize content`）

## 3) 新コピー（差し替え指定）
※「改行」は段落として分けて良い。必要なら <br/> ではなく段落/要素分割を優先。

### 3-1) Hero
- H1：現状の「AIに、臨床の魂を。」は維持
- サブ見出し（新規/差し替え）：
  「医療現場のAI運用を、監査に耐える形へ。」
- 追い文（本文）：
  「ルール・ログ・教育を最小構成で整備し、「導入して終わり」を防ぎます。」
- 補足（小さめの本文・箇条書きでもOK）：
  「ベンダー導入支援／院内SOP整備／リスク点検（監査）／職員研修まで、必要なところだけ段階的に。」
- CTA（主要ボタン文言）：
  1) 「3分で現状チェック（無料）」
  2) 「最短15分で相談する」
- CTA下の一言（あれば）：
  「売り込みではなく、現状整理から始めます。」

### 3-2) セクション：知的資産（見出し/本文）
- 見出し：
  「知的資産を守ることが、AIガバナンスの出発点。」
- 本文（段落 + 箇条書き推奨）：
  段落：
  「AI活用で増えるのは、便利さだけではありません。施設の“守るべき資産”も増えます。」
  箇条書き：
  - 「患者情報・業務データ」
  - 「院内ルール（SOP）と意思決定の根拠」
  - 「プロンプト／テンプレ／運用ノウハウ（属人化しやすい資産）」
  - 「ログ・証跡（説明責任を支える資産）」
  締め：
  「Cursorversは、これらを『ルール × ログ × 教育』で守る運用を設計します。」

### 3-3) セクション：Empower the Individual（英語見出しがある場合）
- 見出し：
  「個人が理解できるルールに落とす。だから、組織で回る。」
- 本文：
  「AI運用が崩れる原因の多くは『ルールが難しい』ことです。
   Cursorversは、院長・医療安全・情シス・現場が同じ地図で動けるように、
   短い院内ルール／チェックリスト／研修として落とし込みます。
   “属人化しない運用”を前提に設計します。」

### 3-4) セクション：AIガバナンスの民主化
- 見出し：
  「大病院級のガバナンスを、中小でも回る最小構成へ。」
- 本文：
  「ガバナンスは『重い仕組み』ではなく、事故を防ぐ運用です。
   まずは最小構成（ルール／ログ／教育）から始め、必要に応じて監査・委員会運用・SaaSへ拡張できます。」

### 3-5) セクション：セルフ診断（3分）
- 見出し：
  「3分で、AI運用リスクを見える化。」
- 説明：
  「いくつかの質問に答えるだけで、いま優先して整えるべき『3つのポイント』と、次の一手が分かります。」
- 取得できるもの（UIが許せば箇条書きで追加、難しければ小さめテキストで追加）：
  - 「リスクの傾向（情報管理／運用ルール／教育／ベンダー依存 など）」
  - 「優先順位つきToDo（最短で効く順）」
  - 「相談が必要な論点の切り分け（院内で済む／外部が必要）」
- CTA：
  「無料でセルフ診断を始める」
  （サブボタンがある場合）「診断結果をもとに相談する」

### 3-6) セクション：48h診断
- 見出し：
  「48時間で、監査目線のToDoを『納品』します。」
- 説明：
  「セルフ診断の結果とヒアリングをもとに、院内で回る形に落とした『運用の叩き台』を提示します。」
- 成果物：
  - 「リスク一覧（重大度×優先度）」
  - 「院内AI利用ルール（叩き台）」
  - 「チェックリスト（運用／ベンダー評価／教育）」
  - 「30日・90日のロードマップ」
- CTA：
  「48h診断について相談する」

### 3-7) セクション：なぜCursorversなのか
- 見出し：
  「“導入”ではなく、“運用”を設計するからです。」
- 3つの理由（カード形式のまま差し替え）
  1) 見出し： 「現場起点」
     本文： 「机上の正論ではなく、院内の制約（人・時間・委員会）を前提に組みます。」
  2) 見出し： 「中立の立場」
     本文： 「ベンダー販売が目的ではありません。複数選択肢を比較し、運用で守ります。」
  3) 見出し： 「証跡を残す」
     本文： 「ルール・ログ・教育を“資産”として残し、説明責任に耐える状態を目指します。」

### 3-8) 3 Core Business Axes（命名の微調整）
- 可能ならカードのタイトル/サブタイトルを以下へ寄せる（UI変更が重いなら本文だけでもOK）
  - 「監査（Audit）」： 「いまの運用を点検し、リスクと改善優先度を可視化します。」
  - 「運用設計（Setup & Policy）」： 「院内ルール・ログ・役割分担を最小構成で整備します。」
  - 「教育（Education）」： 「職員が迷わない教材と研修で、属人化しない運用を作ります。」

### 3-9) 顧問パートナー（Lite/Standard/Premium）
- 見出し（可能なら）：
  「顧問パートナー：運用が積み上がる伴走。」
- 説明（可能なら追加）：
  「月次の定例＋随時レビューで、ルール・ログ・教育が“資産”として増えていく顧問です。」
- 可能なら共通の一言を追加：
  「解約しても、院内に残る成果物（SOP/チェックリスト/研修資料）が資産になります。」

## 4) 実装の推奨（拡張性）
- `landingCopy` の中で、共通語彙（例：governanceCore = "ルール × ログ × 教育"）を定数化し、
  複数箇所で使う（将来のコピー変更が1箇所で済む）
- 段落は配列で持って map 表示すると、将来のセクション追加が容易
  例：hero.description = ["...", "..."] / bullets = ["...", "..."]

## 5) 受け入れ条件（Acceptance Criteria）
- 旧文言（検索ワードに該当）が主要導線から消えている（必要な部分以外は残さない）
- SP/PCで
  - Heroの文章が読みやすい（詰まりすぎない）
  - CTAが押しやすい（幅・折返し・余白）
  - 箇条書きのインデントが崩れていない
- `lint` / `build` が成功
- コミット済み

## 6) 最後に報告してほしいこと（出力）
- 変更したファイル一覧
- どの旧文言をどの新文言に置換したかの要約
- 表示調整のために行ったクラス変更（あれば）
- 実行したコマンド（lint/build）

以上を完了してください。
```

---

### 補足（運用の守備）

* 文章が長くなった分、**SPの折返し**が発生しやすいので、Claude Codeには「段落を分ける」「箇条書きを使う」「max-w調整」までやらせるのが安全です。
* 「監査に耐える」は強いですが、**保証に見えない**ように “目指す/整備する/提示する” を徹底させています（ここは炎上防御）。
